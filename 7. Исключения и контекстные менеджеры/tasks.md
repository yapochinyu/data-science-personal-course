
## Ключевые идеи

1. Исключение — объект-сигнал о нештатной ситуации; оно прерывает выполнение `try` немедленно в точке возникновения — строки ниже места ошибки не выполняются никогда, даже если ошибка потом поймана.
2. Непойманное исключение всплывает вверх по стеку вызовов (C → B → A), пока его не поймает чей-то `except`; если не поймал никто — программа падает с traceback, который читается снизу вверх: внизу сама ошибка, выше — путь к ней.
3. Исключения — классы в иерархии наследования: `except Родитель` ловит и всех наследников (`except OSError` поймает `FileNotFoundError`); `except Exception` ловит все «обычные» ошибки.
4. `KeyboardInterrupt`, `SystemExit`, `GeneratorExit` — наследники `BaseException` напрямую, мимо `Exception`: это намеренно, чтобы `except Exception` не мешал убить программу по Ctrl+C. Голый `except:` ловит и их — поэтому он красный флаг.
5. `except`-блоки проверяются по порядку сверху вниз, срабатывает первый совпавший — поэтому порядок строго от частного к общему; `except Exception` первым делает все ветки ниже мёртвым кодом.
6. `except (TypeError, KeyError):` — кортеж ловит несколько типов одной веткой; `except ... as e` даёт объект ошибки для логирования; молча проглотить ошибку — худшее решение, минимум — залогировать (`logger.exception` добавляет traceback).
7. `else` выполняется, только если `try` прошёл без ошибок; его смысл — держать `try` минимальным: в `try` только опасная строка, продолжение в `else`, чтобы оно случайно не попало под тот же `except`.
8. `finally` выполняется всегда: при успехе, при пойманной ошибке, перед вылетом непойманной, и даже при `return` из `try`/`except` — до фактического возврата значения. `return` внутри `finally` перекрывает любой другой return — антипаттерн.
9. `raise` — бросить исключение самому: валидация входа (лучше упасть сразу с понятным сообщением, чем молча посчитать ерунду); свои классы исключений — наследники `Exception`; голый `raise` в `except` перебрасывает пойманное с исходным traceback; `raise New from e` сохраняет цепочку причин.
10. Проблема, которую решает `with`: `close()` после ошибки недостижим — ресурс утекает; `with` = `try/finally`, упакованный в протокол объекта, с гарантией уборки при любом выходе из блока.
11. Протокол контекстного менеджера: `__enter__()` вызывается при входе, его возвращаемое значение попадает в `as`-переменную; `__exit__(exc_type, exc_val, exc_tb)` вызывается при любом выходе и получает данные об исключении (или три `None`); `return True` из `__exit__` гасит исключение.
12. `@contextmanager` превращает генератор-функцию в контекстный менеджер: код до `yield` — вход, `yield` отдаёт `as`-значение и замораживается на время тела `with`, код в `finally` после `yield` — гарантированный выход.
13. `with` — не только про файлы: транзакции БД, локи, `torch.no_grad()`, временные каталоги, таймеры, смена настроек — общий протокол управления «взял ресурс → гарантированно вернул».
14. EAFP («сделай, потом лови») vs LBYL («сначала проверь»): EAFP идиоматичнее в Python, выигрывает при редких промахах и в многопоточности (нет зазора между проверкой и использованием); для словарей чаще всего лучший вариант — `d.get(key, default)`.
15. Ловить нужно только ошибки, от которых знаешь, как восстановиться: узкий тип + осмысленное действие (лог, дефолт, retry, проброс). Свои баги (опечатки, `NameError`) — чинить, а не оборачивать; программа, которая «не падает», но молча делает ерунду — хуже упавшей.

> Честное (в): `if (val := cache.get(x)) is None: val = compute(x); cache[x] = val` — или классика в две строки. `setdefault(x, compute(x))` страдает той же болезнью, что и get. EAFP-вариант (б) вычисляет compute только в ветке except — лениво.

> [!hint]- Подсказка 3 
> Race condition в (а): между `if x in cache` и `cache[x]` другой поток может удалить ключ → KeyError в «проверенном» коде. EAFP атомарен в смысле «одна операция доступа». Проговори: «проверка и использование — две операции, между ними мир может измениться».

### Задача 14 (идеи 4, 15) — провокация (найди все грехи)

Ревью кода стажёра — найди минимум четыре проблемы и перепиши правильно:

```python
def process_dataset(paths):
    results = []
    for path in paths:
        try:
            f = open(path)
            data = json.load(f)
            results.append(transform(data))
            f.close()
        except:
            pass
    return results
```

> [!hint]- Подсказка 1
> Грехи ищи в трёх местах: как открыт файл, что ловится, что делается с пойманным. И один тонкий — где стоит close относительно возможных ошибок.

> [!hint]- Подсказка 2 
> (1) Голый except ловит даже KeyboardInterrupt — Ctrl+C не работает. (2) pass — замалчивание: битые файлы исчезают бесследно. (3) close не в finally: при ошибке json.load файл течёт. (4) transform под тем же except — его баги тоже проглатываются.

> [!hint]- Подсказка 3 
> Правильно: `with open(path) as f:` внутри try; ловить `(OSError, json.JSONDecodeError)` as e; логировать путь и ошибку; transform — вне try или под отдельной обработкой. Устно закрой: «ловлю узкий ожидаемый тип, пойманное логирую, свои баги не прячу».

### Задача 15 (идеи 2, 8, 9 — интеграционная) — провокация + предсказание

Не запуская, выпиши точный порядок печати и финальный исход (что вернётся / что упадёт):

```python
def inner():
    try:
        print('inner try')
        raise KeyError('k')
    finally:
        print('inner finally')

def outer():
    try:
        inner()
        print('after inner')
    except KeyError as e:
        print(f'outer caught {e}')
        return 'handled'
    finally:
        print('outer finally')

print(outer())
```

> [!hint]- Подсказка 1
> В inner нет except — KeyError не будет пойман там. Но finally в inner всё равно выполнится. Когда именно — до или после того, как исключение покинет inner?

> [!hint]- Подсказка 2 
> Порядок: inner try → inner finally (перед вылетом!) → исключение всплывает в outer → 'after inner' пропущен → outer caught → перед return сработает outer finally.

> [!hint]- Подсказка 3 
> Полный вывод: inner try, inner finally, outer caught 'k', outer finally, handled. Это вся тема в одном примере: всплытие по стеку + finally при непойманной ошибке + finally при return. Проговори каждую строку с объяснением — идеальная устная защита.
