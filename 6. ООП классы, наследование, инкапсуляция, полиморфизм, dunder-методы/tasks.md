
## Ключевые идеи

1. Класс — чертёж нового типа (данные + поведение), экземпляр — конкретный объект по чертежу; один класс — много независимых экземпляров, каждый со своими данными и общими методами.
2. `self` — не ключевое слово, а первый параметр метода: `obj.method(x)` эквивалентно `Class.method(obj, x)`; self нужен, чтобы метод знал, с данными какого объекта работать. Отсюда ошибка «takes 1 positional argument but 2 were given» — забыт self в определении.
3. `__init__` — не создание объекта, а инициализация уже созданного: Python выделяет объект, вызывает `__init__(новый_объект, ...)`, где через `self.attr = ...` вешаются атрибуты экземпляра; `__init__` возвращает `None`.
4. Атрибут экземпляра (`self.x` в `__init__`) — свой у каждого объекта; атрибут класса (в теле класса вне методов) — один на всех. Изменяемый атрибут класса (`toys = []`) — мина: все экземпляры мутируют один общий объект (та же механика, что изменяемый дефолт).
5. При чтении `obj.attr` Python ищет сначала у экземпляра, потом у класса; присваивание `obj.attr = ...` (в т.ч. через `+=`) создаёт атрибут экземпляра, затеняющий классовый, — атрибут класса при этом не меняется.
6. В Python нет private: `_x` — соглашение «внутреннее, не трогай»; `__x` — name mangling (переименование в `_ClassName__x`) для защиты от конфликта имён при наследовании, а не от доступа. Инкапсуляция — на уровне договорённостей.
7. `@property` делает метод, выглядящий как атрибут: геттер для чтения, `@attr.setter` для валидации при присваивании, вычисляемые поля; позволяет менять внутренности, не ломая внешний API.
8. Наследование: ребёнок получает все методы и атрибуты родителя, может добавить свои и переопределить (override) родительские; поиск метода идёт от ребёнка вверх.
9. Если у ребёнка свой `__init__`, родительский сам НЕ вызывается — нужен `super().__init__(...)`; без него родительские атрибуты не создадутся и объект будет битым (`AttributeError` при обращении).
10. `isinstance(x, Cls)` учитывает наследование (наследник проходит проверку родителя), `type(x) == Cls` — только точное совпадение; в полиморфном коде правилен `isinstance`.
11. Полиморфизм — один интерфейс, разные реализации: код вызывает `model.fit(X, y)` в цикле по разным моделям sklearn, не зная конкретного типа.
12. В Python полиморфизм работает через утиную типизацию: наследоваться от общего базового класса не обязательно — достаточно иметь нужные методы; расплата — ошибки только в runtime (`AttributeError`).
13. Dunder-методы встраивают класс в язык: Python сам переводит `len(x)` → `__len__`, `x[i]` → `__getitem__`, `a + b` → `__add__`, `a == b` → `__eq__`, `print` → `__str__`/`__repr__`, `for` → `__iter__` (или `__getitem__`), `x()` → `__call__`.
14. `__repr__` — для разработчика (как воссоздать объект), `__str__` — для пользователя; если есть только `__repr__` — используется везде; в списках всегда `__repr__`. Правило: `__repr__` определяй всегда.
15. Без своего `__eq__` оператор `==` сравнивает по идентичности (как `is`); определяя `__eq__`, задаёшь свою логику равенства (и теряешь хешируемость по умолчанию).
16. PyTorch Dataset — это ровно `__len__` (сколько примеров) + `__getitem__` (выдай пример №i): DataLoader по ним считает батчи и лениво запрашивает элементы.
17. ООП оправдан для сущностей с состоянием, живущим во времени (модель с весами); для скриптов и чистых функций — оверкилл; для «пачки данных без поведения» — `@dataclass` (автогенерация `__init__`, `__repr__`, `__eq__`).

> Внутри loader: `batch = [dataset[i] for i in range(start, min(start + batch_size, len(dataset)))]` и `yield batch`. Ни одного обращения к `dataset.lines` — только через dunder-интерфейс.

> [!hint]- Подсказка 3 
> Ответ на «почему»: loader программирует против интерфейса (`__len__` + `__getitem__`), а не против реализации — датасет может держать данные в списке, файле или базе, loader не изменится. Это и есть контракт PyTorch DataLoader ↔ Dataset. Бонус: заметь, что `for x in dataset` заработал автоматически — почему?

### Задача 13 (идея 14) — провокация + предсказание

Не запуская, выпиши вывод:

```python
class A:
    def __repr__(self):
        return 'repr_A'

class B:
    def __str__(self):
        return 'str_B'

a, b = A(), B()
print(a)
print(b)
print([a, b])
print(f'{a} and {b}')
```

> [!hint]- Подсказка 1 
> Правила: print и f-строки используют `__str__`, но если его нет — падают обратно на `__repr__`. Контейнеры (список) — всегда `__repr__` элементов.

> [!hint]- Подсказка 2
> У A есть только repr → везде repr_A. У B есть только str → print даст str_B, а в списке... что покажет repr по умолчанию?

> [!hint]- Подсказка 3 
> Третий print: `[repr_A, <__main__.B object at 0x...>]` — у B нет своего `__repr__`, наследуется дефолтный от object с адресом. Вывод-правило: «определяй `__repr__` всегда — он подстрахует всё; `__str__` — опционально».

### Задача 14 (идея 15) — код руками + каверза

Дан класс:

```python
class User:
    def __init__(self, user_id, name):
        self.user_id = user_id
        self.name = name
```

(а) Сейчас `User(1, 'Anna') == User(1, 'Anna')` даёт False — почему? Добавь `__eq__` (равенство по user_id). (б) После этого выполни `{User(1, 'Anna'): 'admin'}` — что произойдёт и почему? Свяжи с темой хеширования из первой темы. (в) Почини, добавив согласованный `__hash__`.

> [!hint]- Подсказка 1 
> (а) Дефолтный `__eq__` от object сравнивает как `is` — два разных объекта в памяти. (б) Python имеет правило: определил `__eq__` — `__hash__` автоматически становится None.

> [!hint]- Подсказка 2 
> (б) `TypeError: unhashable type: 'User'`. Причина из темы 1: инвариант `a == b → hash(a) == hash(b)` — старый хеш (по id объекта) сломал бы его после нового `__eq__`, поэтому Python отключает хеширование принудительно.

> [!hint]- Подсказка 3 
> (в) `def __hash__(self): return hash(self.user_id)` — хеш по тому же полю, по которому равенство: инвариант соблюдён. Устно: почему хешировать по изменяемому полю (name, если его меняют) опасно для dict?

### Задача 15 (идея 13) — код руками (`__call__`, мост к DL)

Напиши класс `Scaler`: конструктор принимает `mean` и `std`; объект вызывается как функция — `scaler(x)` возвращает `(x - mean) / std`; метод `fit(data)` вычисляет mean и std по списку и возвращает `self` (чтобы работала цепочка `Scaler().fit(data)(x)`). Продемонстрируй цепочку. Устно: какой dunder делает объект вызываемым и где ты уже видел этот паттерн в PyTorch?

> [!hint]- Подсказка 1 
> Вызываемость — `__call__(self, x)`. Цепочка работает, потому что fit возвращает... что?

> [!hint]- Подсказка 2 
> `return self` в конце fit — тогда `Scaler().fit(data)` — это объект, и к нему сразу применимы скобки `(x)` → `__call__`. std: корень из среднего квадратов отклонений (`** 0.5`).

> [!hint]- Подсказка 3 
> PyTorch: `output = model(input)` — это `__call__` класса `nn.Module`, который внутри вызывает твой `forward` (плюс хуки). Поэтому пишешь forward, а вызываешь объект. Проговори это — частый вопрос «почему model(x), а не model.forward(x)».

### Задача 16 (идея 17) — код руками + рассуждение

Есть класс:

```python
class TrainConfig:
    def __init__(self, lr, epochs, batch_size):
        self.lr = lr
        self.epochs = epochs
        self.batch_size = batch_size
```

Перепиши его через `@dataclass` со значениями по умолчанию (`lr=0.01`, `epochs=10`, `batch_size=32`). Продемонстрируй, что бесплатно появились `__repr__` и `__eq__`. Каверза: добавь в dataclass поле `callbacks: list = []` — что скажет Python и почему? Как правильно задать изменяемый дефолт в dataclass?

> [!hint]- Подсказка 1 
> `@dataclass` + аннотации типов вместо `__init__`. Сравни два одинаковых конфига через `==` — до и после.

> [!hint]- Подсказка 2 
> `callbacks: list = []` даст `ValueError: mutable default ... use default_factory` — dataclass знает про ловушку изменяемого дефолта и запрещает её на уровне объявления!

> [!hint]- Подсказка 3 
> Правильно: `from dataclasses import field` и `callbacks: list = field(default_factory=list)` — фабрика вызывается при каждом создании, каждый конфиг получает свой список. Устная связка: «dataclass — это язык, который выучил урок из `def f(x, lst=[])`».
